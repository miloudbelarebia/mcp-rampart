The blind spot

Every MCP server ships the same blind spot.

Gateways watch the wire. Config scanners watch the file. Nobody is watching the routes you're about to expose to an LLM — from inside your code.

Without mcp-rampart

● Wire-only
# your FastAPI app boots, mounts /mcp, ships. @app.delete("/api/admin/users/{id}") async def del_user(id: int): ... deletes the user. # LLM client connects, sees a `del_user` tool. # No-one warned you.
  • ✕/admin and /auth endpoints exposed silently
  • ✕Pydantic responses leaking email/phone into LLM context
  • ✕Tools without docstrings — the LLM guesses
  • ✕No filter on the args your tools accept

With mcp-rampart

● Code-side
# 3 lines after your routes. rampart = MCPRampart(app) rampart.audit().print_text() rampart.enable_guardrails(policy="block") 🔴 CRITICAL /api/admin/... exposed to LLM → server refuses to start.
  • ✓Refuses to boot on CRITICAL findings (auth, admin)
  • ✓Detects PII fields in response_model
  • ✓Flags missing docstrings, untyped params, wildcard responses
  • ✓Scans every tools/call for 27 injection patterns

Refuses to start on CRITICAL

The audit's has_blockers() short-circuits boot. Your CI fails before the dangerous server ships — not after.

report = rampart.audit() if report.has_blockers(): report.print_text() raise SystemExit(1)

Why this exists

Built because the wire-tools can't see your code.
Open-sourced because the problem keeps repeating.

I was shipping a FastAPI app as an MCP server. A friend asked: "you sure your /api/admin/* isn't in the tool list a language model sees?" — and I had to actually grep to find out. That's the moment I realised the wire-side gateways and firewalls everyone was shipping couldn't answer that question. They didn't see the decorator. So I wrote the linter I wished I had: rampart.audit() at startup, rampart.enable_guardrails() at runtime, refuse-to-start on CRITICAL, MIT, no telemetry.

FAQ

Common questions.

Why not just be a generic MCP proxy that works with any framework?

Because the audit needs to read your code — Pydantic response models, @app.delete decorators, type hints, docstrings. A proxy on the wire only sees the JSON-RPC payload. It can't tell you "you're about to expose your admin route to an LLM" because that knowledge lives three steps upstream from the wire.

Is this the same as pipelock / casbin-gateway / hyprmcp / apisec mcp-audit?

No. They live at layers 2, 3, and 5 (see the diagram above). mcp-rampart lives at layer 4. They protect the operator and the user; mcp-rampart protects the server author. You usually want more than one of them.

Do I still need a gateway / firewall if I use mcp-rampart?

Probably yes. mcp-rampart catches code-side issues and runtime injection. A gateway adds auth, rate-limiting, and network policy. They're complementary, not substitutes.

What happens if mcp-rampart blocks a legitimate call?

The MCP client receives an isError: true response with the diagnostic under _mcp_rampart (matched patterns + reason). Switch to policy="alert" while tuning. Per-route bypass via rampart.exclude("/path/*").

Can I add my own rules?

Yes. Subclass Auditor or InjectionDetector and pass it via custom_detector=. A first-class plugin / decorator API ships in v0.6.

Flask? Django? Node.js?

FastAPI today. Node.js (Hono / Express / Fastify) is v0.5 — that's where most new MCP servers ship in 2026. Flask + Django are v0.6.

Does it call home?

No. No network calls, no telemetry, no SaaS upsell. Pure regex and stdlib introspection. MIT licensed.

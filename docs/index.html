Section II — the story

I was three commits away from shipping a FastAPI app as an MCP server. A friend asked the question I should have asked myself: "are you sure your admin routes aren't in the tool list a language model will see?"

I went to grep. The answer was no. Or rather — yes, they were exposed. By default. Because nothing in the FastAPI → MCP pipeline asks "should this particular endpoint be visible to a Claude or a ChatGPT or a Cursor?" The pipeline only asks "is this a valid route?"

I looked at what existed. pipelock watches the wire on the client side. casbin-gateway enforces policy in a proxy. apisec/mcp-audit scans my ~/.mcp.json for leaked secrets. All useful. None of them could read my @app.delete decorator three steps upstream from the wire.

So I built the linter I wished I had. It reads app.routes. It reads Pydantic response_models. It flags routes whose path matches /auth/ or /admin/ at CRITICAL. It refuses to start the server when there's even one CRITICAL. It scans every tools/call at runtime for the kind of patterns LLMs see and listen to.

I tested it on the official Auth0 example of tadata-org/fastapi_mcp — the most popular library in the space, 11.9k stars. The example, used as-is, exposes /oauth/authorize, /oauth/register, and the OAuth metadata endpoint to LLM clients. mcp-rampart catches all three. The server refuses to start. The case study is on the repo.

That's it. Three lines, MIT, no telemetry. Read every line of the source — it's about 1,500. Submit a pattern that should have been caught, or a check that the audit missed. The issue template's already there.

Section III — the field

Five layers of MCP security. mcp-rampart is at four.

Every shipping project in MCP security solves a different problem at a different layer. Here is the map, with the projects that hold each ground.

  1. I

    The language model itself model providers

    Hallucinations, jailbreaks at the model level. Out of scope for everyone except the labs that train the weights.

  2. II

    The MCP client on the wire

    Watches what the language model decides to call. Egress control, DLP, agent firewall. Sits between the LLM-running app and the network.

    pipelock · mcp-firewall · SecretiveShell/MCP-Bridge
  3. III

    The gateway in front of your server on the wire

    Who is allowed to talk to this MCP server, with what auth, with what rate limit. A reverse proxy with a policy engine.

    apache/casbin-gateway · hyprmcp/mcp-gateway
  4. IV

    The MCP server itself you are here

    Did the code I'm about to ship accidentally expose an admin route, leak PII in a response, accept a prompt-injection in a tool argument? The proxy can't answer because the answer lives in the decorator three steps upstream.

    mcp-rampart — this project
  5. V

    The user's MCP config on disk

    Am I about to install a malicious MCP server on my own machine? Scans ~/.mcp.json and downloaded server packages for unsafe patterns.

    apisec-inc/mcp-audit · ModelContextProtocol-Security/mcpserver-audit
Section IV — the catalogue

Twenty-seven patterns. Three confidences. No model.

No classifier, no embedding, no API call. A regex catalogue you can read in 200 lines of source. Submit a pattern that should be here — the issue template is missing_pattern.md.

HIGH — block on sight

  • instruction_override "ignore previous instructions"
  • role_override "you are now …"

  • system_impersonation "SYSTEM: …"

MEDIUM — block on stack

  • system_reference "system prompt"
  • role_play "act as …", "pretend to be …"

Section V — questions

Common objections.

Why not just be a generic MCP proxy that works with any framework?

Because the audit needs to read your code — Pydantic models, @app.delete decorators, type hints, docstrings. A proxy on the wire only sees the JSON-RPC payload. It can't tell you "you're about to expose your admin route to a language model" because that knowledge lives three steps upstream from the wire.

Is this the same as pipelock / casbin-gateway / hyprmcp / apisec mcp-audit?

No. They live at layers 2, 3, and 5 (see the field manual above). mcp-rampart lives at layer 4. They protect the operator and the user; mcp-rampart protects the server author. You usually want more than one of them.

Do I still need a gateway / firewall if I use mcp-rampart?

Probably yes. mcp-rampart catches code-side issues and runtime injection. A gateway adds auth, rate-limiting, network policy. Complementary, not substitutes.

What happens if mcp-rampart blocks a legitimate call?

The MCP client receives an isError: true response with the diagnostic under _mcp_rampart (matched patterns + reason). Switch to policy="alert" while you tune. Per-route bypass via rampart.exclude("/path/*").

Can I add my own rules?

Yes. Subclass Auditor or InjectionDetector and pass it via custom_detector=. A first-class plugin / decorator API ships in v0.6.

Flask? Django? Node.js?

FastAPI today. Node.js (Hono / Express / Fastify) is v0.5 — that's where most new MCP servers ship in 2026. Flask + Django are v0.6.

Does it call home?

No network calls. No telemetry. No SaaS upsell. Pure regex and stdlib introspection. MIT.
